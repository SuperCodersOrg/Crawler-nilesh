# Daily Journal – 9 July 2026

## Section 1 — Specific Bug

### Bug 1 – URL Normalizer

**Runtime Observation**

Input:

```
HTTP://Example.COM:80/a/./b/../c//index.html#abc
```

Expected Output:

```
http://example.com/a/c/index.html/
```

Actual Output:

```
http://example.com:80index.html/
```

**Root Cause**

The normalizer incorrectly checked for port `8080` instead of the default HTTP port `80`, preventing removal of the default port during canonicalization.

---

### Bug 2 – Path Reconstruction

**Observed Output**

Only the final directory component of the normalized path was preserved.

Example:

```
/index.html/
```

instead of

```
/a/c/index.html/
```

**Root Cause**

The reconstruction loop reassigned the entire result string during every iteration instead of appending each directory component.

---

### Bug 3 – HashMap Key Enumeration

While implementing the custom `Set`, it became clear that the existing `HashMap` did not provide any method to retrieve every stored key. This prevented iteration over the complete collection of stored elements in both the crawler and the Redis-lite project.

---

### Bug 4 – HashMap Remove Operation

**Compiler Error**

Compilation failed while implementing `Set::remove()` because the linked list accepted an index for deletion, whereas the `HashMap` attempted to remove an entire `Pair` object directly.

## Section 2 — Failed Attempts

### URL Normalizer

Initially assumed that correcting the default port comparison would completely solve the normalization issue. After fixing the port check, testing revealed that the normalized path was still incorrect because the reconstruction logic discarded previously processed directories. This showed that multiple independent bugs existed inside the normalization pipeline.

### HashMap

My first implementation of `getkeys()` treated each bucket as if it stored only one element. After reviewing the collision handling mechanism, I realized that each bucket actually contains a linked list due to separate chaining. The implementation was redesigned to traverse every node inside every bucket.

### Set Implementation

During deletion, I initially attempted to remove the entire `Pair` object directly from the linked list. Since the linked list API only accepted an index-based removal function, the implementation had to be redesigned to first locate the correct node and then remove it using its index.

## Section 5 — Learning Reflection

Today's work reinforced that building higher-level data structures depends heavily on the correctness and flexibility of lower-level implementations. While implementing the crawler, I realized that URL normalization involves several independent transformations rather than a single operation, and each stage must preserve correctness before passing data to the next stage.

Implementing `getkeys()` also improved my understanding of collision handling in hash tables. Although the bucket array appears simple, every bucket can represent an entire linked list of elements because of separate chaining. Ignoring collision chains results in incomplete traversal and incorrect behavior.

Finally, implementing the custom `Set` highlighted the importance of interface compatibility between dependent data structures. Even when individual components work correctly, mismatched APIs can prevent successful integration. Updating the `HashMap` deletion logic to match the linked list interface made the overall data structure library more reusable and consistent for future projects.
