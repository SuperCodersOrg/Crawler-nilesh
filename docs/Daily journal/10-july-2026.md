# Daily Journal – 10 July 2026

## Section 1 — Specific Bug

### Bug 1 – Infinite Loop During HTML Parsing

While testing the initial parser implementation, the crawler repeatedly processed the same portion of the HTML without progressing further.

**Root Cause**

The parsing index was not being advanced correctly after extracting a URL. As a result, the parser repeatedly scanned the same substring, resulting in an infinite loop.

---

### Bug 2 – Linker Error During Integration

**Compiler Output**

```text
undefined reference to HtmlParser::parseHtml(...)
```

**Root Cause**

Although the parser implementation had been completed, `HtmlParser.cpp` was not included in the CMake build configuration. The compiler successfully found the declarations, but the linker could not locate the implementation.

---

### Bug 3 – Multiple Class Definition Error

**Compiler Output**

```text
error: redefinition of 'class Frontier'
```

**Root Cause**

While integrating the crawler components, the same header file was included through multiple dependency paths. Since the header did not contain proper include guards, the compiler processed the class definition more than once.

**Resolution**

Added `#pragma once` to the required header files to prevent multiple inclusion.

---

## Section 2 — Failed Attempt

Initially, the HTML parser extracted URLs by repeatedly creating new substrings during parsing. Although this approach worked for simple cases, it introduced unnecessary copies and caused index management problems, including infinite loops when the parsing position was not updated correctly.

During crawler integration, I also assumed that completing the implementation files would be sufficient for successful compilation. Later, I realized that every newly created source file must also be added to the CMake configuration; otherwise, linker errors occur even though the code compiles successfully.
