### 2. SQLite Integration

A significant portion of today's work focused on understanding and integrating **SQLite** into the project.

The objective behind using SQLite was to move away from temporary in-memory storage and provide persistent storage for downloaded webpages.

During this process, I learned about:

* Creating and opening SQLite databases.
* Managing database connections.
* Designing a database-backed storage layer for webpages.
* Storing structured webpage information inside database tables.
* Retrieving stored data using SQL queries.
* Closing database connections safely after execution.

The `PageStorage` component was redesigned around SQLite so that downloaded pages can be stored permanently instead of existing only during program execution.

Each stored page is intended to contain information such as:

* URL
* Rendered HTML
* Crawling depth

This approach improves scalability and also makes the stored data reusable for future modules such as the Indexer.

### 3. Started URL Normalizer Implementation

Today's major development task was beginning the implementation of the **URL Normalizer** inside the `Frontier` component.

The purpose of the normalizer is to convert different representations of the same URL into a consistent format before they are inserted into the crawling frontier. This helps prevent duplicate crawling and simplifies URL comparison.

Before writing the implementation, I studied the complete structure of URLs based on the URI standard.

The following URL components were analyzed:

* Scheme (HTTP / HTTPS)
* User information
* Host
* Port
* Path
* Query parameters
* Fragment

I also studied different types of hosts, including:

* Domain names
* IPv4 addresses
* IPv6 addresses

Additionally, I explored several normalization rules that are commonly used by production web crawlers, including:

* Converting the scheme to lowercase.
* Converting the hostname to lowercase.
* Removing default ports (`80` for HTTP and `443` for HTTPS).
* Removing URL fragments (`#...`).
* Removing duplicate slashes.
* Resolving `.` (current directory) segments.
* Resolving `..` (parent directory) segments.
* Understanding how relative URLs are converted into absolute URLs.
* Identifying tracking query parameters that may be removed during normalization.

As part of this module, I implemented the initial parsing function:

```cpp
string parseScheme(string source);
```

This function extracts the scheme (protocol) from a URL and serves as the first building block of the complete URL normalization pipeline.
